Templates for the two core planning documents generated during PRD_ARCH phase.

## PRD Template (Product Requirements Document)

This template is filled in and generated by the Orchestrator during Phase 3 based on information collected in Phase 0.
The generated result uses a "thin entry + modules" structure:

- `docs/PRD.md`: entry point and stable links
- `docs/prd/*.md`: actual content and Milestone definitions

`bun .harness/init.ts --from-prd` will prioritize reading `docs/prd/*.md`; if the module directory doesn't exist, it falls back to the single `docs/PRD.md`.

### PRD Update Process

When new requirements come in:

1. If the scope belongs to the current delivery phase, update the current phase's Milestone list and run `bun harness:sync-backlog`
2. If the scope belongs to a later delivery phase, add it to a draft phase bucket (e.g. `V2`) instead of activating it immediately
3. After the current launch phase passes deploy review, approve the next draft phase with `bun harness:approve --phase V[N]`, then promote it with `bun harness:stage --promote V[N]`

**The PRD is the single source of truth for requirements.** Any feature not written into the PRD will not be executed by the Orchestrator.

## Architecture Template

Architecture uses a "thin entry + modules" structure:

- `docs/ARCHITECTURE.md`: entry point and stable links
- `docs/architecture/*.md`: system overview, structure, dependency rules, validation strategy, decisions
- `docs/architecture/versions/*.md`: version snapshots captured when a new delivery stage is promoted

Treat the directory names, manifests, and commands below as reference defaults, not fixed requirements. The final Architecture must follow the confirmed stack in the PRD and the detected project toolchain.

---

> **Version**: v1.0
> **Last updated**: [DATE]
> **Related documents**: [PRD.md](./PRD.md) | [../AGENTS.md](../AGENTS.md)

---

[CLIENT]          [EDGE/CDN]       [SERVER]          [INFRA]

## 2. Directory Structure

[PROJECT_NAME]/
├── AGENTS.md                ← Codex instructions (identical to CLAUDE.md)
├── CLAUDE.md                ← Claude Code instructions
├── apps/                    ← Product surfaces (web / ios / cli / agent / desktop)

│   ├── prd/, architecture/, adr/
├── src/
│   ├── types/               ← Layer 1: Shared contracts / schemas
│   ├── config/              ← Layer 2: Runtime configuration
│   ├── lib/                 ← Layer 3: Utilities / adapters
│   ├── services/            ← Layer 4: Business logic / use cases
│   └── app/                 ← Layer 5: Delivery surface
├── tests/
│   ├── unit/, integration/, e2e/
└── .github/workflows/

---

## 3. Dependency Direction (Enforced by G5)

types → config → lib → services → app

Higher layers may reference lower layers; lower layers must never reference higher layers.
Circular dependencies are prohibited. CI tests automatically fail on violations.

---

## 4. Layer Responsibilities

- **types/**: Shared contracts, schemas, DTOs — no business logic
- **config/**: Environment and runtime configuration loading
- **lib/**: Utility helpers and infrastructure adapters
- **services/**: Core business logic
- **app/**: Delivery-layer code only — orchestration, no business logic

---

## 5. Testing Strategy

| Layer | Test Type | Target Coverage |
|-------|-----------|-----------------|
| types | Static checks | 100% critical contracts |
| lib | Unit tests | > 80% |
| services | Unit + Integration | > 70% |
| app | Integration + E2E | > 50% |

---

  └── fix/[issue-id]      ← Hotfix
